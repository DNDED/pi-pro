# Decisions — pi-pro v0.7.0

## Memory at Scale release

### New `packages/embeddings` (top of dep stack)
- **Why:** v0.7.0 needs embeddings in 3 different packages (memory-store, codebase-index, context-manager). Centralizing the provider abstraction in one package avoids duplication and gives a single switch to flip between Anthropic Voyage / OpenAI text-embedding-3 / opencode-go / null (BM25-only).
- **Decision:** `packages/embeddings` exposes `EmbeddingsProvider` interface + 4 implementations. Other packages depend on `@pi/embeddings`, not on the provider SDKs directly.
- **Captured:** spec §4.1 + `packages/embeddings/src/{types,anthropic,openai,opencode-go,null}.ts`.

### Embeddings: provider API, not local model
- **Why:** Local models (MiniLM, BGE) require a download, 100MB+ on disk, and 50-100ms per query. For v0.7.0 we ship faster and skip the dep. OpenAI text-embedding-3-small is $0.02/M tokens — essentially free for our use case. Anthropic Voyage-3 is comparable. opencode-go passthroughs the user's existing provider.
- **Decision:** Default to OpenAI text-embedding-3-small (cheapest, widely available). Override via `PROMYRA_EMBEDDINGS=anthropic|openai|opencode-go|null`. If no API key, auto-fallback to NullEmbeddings (BM25 only).
- **Captured:** spec §4.1 + §6 env flags.

### Storage: Global SQLite + per-project markdown
- **Why:** v0.7.0 memory is cross-session. Two layers: a global SQLite DB at `~/.pi-pro/memory.db` for fast structured search, plus a per-project markdown view in `memory/` for human readability and git tracking. The markdown is the source of truth for the project; SQLite is the cache.
- **Decision:** Schema: `chunks(id, project, source, role, text, embedding BLOB, ts)`. Project markdown is auto-indexed on disk-write; out-of-band edits are re-indexed on next access.
- **Captured:** spec §4.2.

### No sqlite-vss for v0.7.0
- **Why:** sqlite-vss is a native extension; it requires building against the host's SQLite version. Brute-force cosine over 10k chunks is <100ms in JS. Premature optimization.
- **Decision:** v0.7.0 ships with brute-force cosine search. If users hit >10k chunks per project, add sqlite-vss in v0.8.0.
- **Captured:** spec §4.2 + `packages/memory-store/src/query.ts`.

### Compression: hybrid (extractive first, LLM fallback)
- **Why:** Extractive compression is free (no LLM call) and preserves verbatim tool results, which are the most useful parts of context. LLM summarization is expensive and can hallucinate. But sometimes extractive isn't enough (e.g. 200-tool-call conversation).
- **Decision:** Pipeline: (1) drop oldest 50% of tool results; (2) drop oldest 30% of non-system messages; (3) if still over budget, run LLM summarization on the dropped window. LLM call is gated and only runs when extractive insufficient.
- **Captured:** spec §4.3.

### Codebase index: both regex + embeddings
- **Why:** Regex (from v0.5.0 repo-map) gives exact symbol names; embeddings give semantic matches ("find code that handles authentication" → AuthService, login, JWT). Neither alone is sufficient.
- **Decision:** Hybrid search: regex exact-match scores first; vector similarity fills remaining slots; path filter is applied. Wraps v0.5.0 `@pi/repo-map` rather than replacing it.
- **Captured:** spec §4.4.

### Auto-summarization triggers: adaptive (3 modes, OR)
- **Why:** Single-mode triggers are wrong: token-threshold alone misses long-but-cheap conversations; turn-count alone misses short-but-token-heavy ones. Three modes OR'd together covers all cases.
- **Decision:** Triggers: (1) token threshold (75% soft-warn, 90% hard-trigger); (2) turn count (every 20 turns); (3) cost cap (configurable per-session, default $0.50). OR logic: any one fires. Soft-warn displays badge but doesn't compress; hard-trigger compresses.
- **Captured:** spec §4.3.

### `/btw`: side-question channel
- **Why:** Users often want to ask "btw, what's X" without polluting the main conversation history. Claude Code has this. Without it, side questions bloat the main context window and get included in future LLM calls.
- **Decision:** `/btw <question>` is a separate LLM call that returns answer to the user only — never enters the main conversation. Implementation: `LlmWorker.runBtw(question)` returns a string; TUI `BtwPrompt` renders input + ephemeral response above the prompt.
- **Captured:** spec §4.3 + §7 TUI surface.

### TUI ContextBudget: live bar in Footer
- **Why:** Users need to see context usage growing in real time. Otherwise v0.7.0 silently hits the 90% threshold and starts compressing without warning.
- **Decision:** New `ContextBudget` component in `packages/tui-pro`. Renders `ctx:84k/200k (42%) [w]` in the Footer. Green < 75%, yellow 75-90% (soft-warn), red > 90% (hard). New `/context` slash command shows full breakdown by category.
- **Captured:** spec §4.5 + §7 TUI surface.

### Live LLM bench still deferred
- **Why:** v0.7.0 adds 4 packages, 50+ tests, and several integrations. Live LLM bench with attribution adds another 50+ LLM calls and requires API key configuration. Sid's environment has no API key configured.
- **Decision:** v0.7.0 ships with the test suite + mocked bench stubs (per v0.5.0/v0.6.0 pattern). Live LLM bench attribution run is a follow-up session.
- **Captured:** spec §11 + plan Task 8.
